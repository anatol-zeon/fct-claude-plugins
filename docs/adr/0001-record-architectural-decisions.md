# 1. Record architectural decisions

Date: <YYYY-MM-DD>

## Status

Accepted

## Context

Проекту нужен лёгкий, версионируемый способ фиксировать архитектурные решения: что выбрали, почему, какие альтернативы были рассмотрены и почему отклонены. Без этого через 3-6 месяцев решения пересматриваются с нуля, а тех-долг становится невидимым.

## Decision

Используем ADR в формате [MADR](https://adr.github.io/madr/) (упрощённый вариант). Каждое решение — отдельный файл `docs/adr/NNNN-<kebab-title>.md`.

Обязательные секции: **Status**, **Context**, **Decision**, **Consequences**. Опционально **Alternatives considered** при спорных выборах.

## Consequences

- Каждое архитектурное решение фиксируется сразу, а не "потом".
- PR, меняющий архитектуру, обязан содержать новый ADR или обновление существующего.
- Отменённое решение помечается `Status: Superseded by ADR-NNNN`, но файл не удаляется.

## Alternatives considered

- **Запись в Notion/Confluence** — отвергнуто: нет версионирования, рвётся связь "код ↔ решение".
- **Только коммит-сообщения** — отвергнуто: коммиты фрагментарны, нет единого места "почему".
